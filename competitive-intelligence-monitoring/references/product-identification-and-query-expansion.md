# 商品识别与查询扩展

## 输入与识别

接受商品链接、ASIN/SKU/商品 ID、店铺链接、商品图片/截图、自然语言描述或品类名称。先提取自有商品卡，再开始检索；不得把平台推荐结果反向当成自有商品事实。

| 输入 | 处理 | 风险控制 |
|---|---|---|
| 商品链接或 ID | 读取标题、品牌、属性、价格、图片、类目和变体 | 页面不可访问时保留失败状态，改用用户输入或截图 |
| 店铺链接 | 识别商品矩阵和最可能目标商品 | 多个候选会改变结果时列出候选供确认 |
| 图片或截图 | 识别可见形态、用途、结构、场景和可能品类 | 不从外观断言材质、性能、认证、销量或目标客群 |
| 自然语言描述 | 提取品类、任务、功能、客群、场景、价格带和差异点 | 将未核验字段标为用户输入或分析假设 |
| 品类名称 | 建立品类级候选池 | 明确不是单品级竞争结论 |

## 最小追问

- 高置信：平台、国家和商品明确，直接执行。
- 中置信：核心品类和任务明确，带假设执行并在结果开头提示。
- 低置信：存在多个可能品类或目标商品，只追问最能缩小竞品池的一个问题。
- 未指定国家但平台只有一个合理站点或上下文已明确时继承；否则补问国家。
- 不要求用户一次填写所有字段；没有决策影响的缺口保留为 `null`。

## 五组查询

| 查询组 | 目的 | 示例结构 |
|---|---|---|
| 核心品类词 | 找直接竞品 | 标准品类名、平台常用名、同义词 |
| 功能需求词 | 找任务重叠商品 | 痛点 + 功能、结果 + 产品 |
| 使用场景词 | 找场景型竞品 | 场景/地点/时段 + 产品或任务 |
| 目标人群词 | 找细分定位 | 人群 + 产品或问题 |
| 替代方案词 | 找不同形态替代品 | 同一核心任务的其他技术或商品形态 |

同时检查核心词 SERP、类目榜单、广告位、相关推荐、店铺矩阵、社媒/内容和供应链反向线索。记录 `query`、平台、市场、发现位置、采集时间和展示条件。

## 扩展与收敛

- 结果过少：扩展同义词、功能词、场景词、替代方案和邻近价格带。
- 结果过宽：按核心任务、客群、场景、价格带、规格或关键属性收紧。
- 搜索停止：连续两轮扩展未发现新的高相关 T1/T2，且核心词、类目、广告位与至少一个补充渠道均已覆盖；记录未覆盖渠道。
- 不以结果数量证明覆盖完整；报告搜索范围、截止时间和召回限制。

## 异常处理

- 链接失效：保留 URL 和失败时间，不编造页面字段。
- 图片模糊：给出最多 3 个候选品类和区分问题。
- 店铺商品过多：按用户描述、图片或近期上下文筛出候选，不擅自锁定。
- 平台无法访问：使用可核验替代来源时说明来源差异；否则输出补证清单并降低置信度。

## 模块执行协议

### 对象与基线

- 锁定我方对象、竞品实体、商品/变体、店铺、平台、国家、展示条件和采集时间。
- 记录检索词、发现渠道、纳入/排除理由、覆盖停止条件和未覆盖面。
- 单次快照只能证明当时可见状态；趋势至少需要三个可比快照。

### 信号与机制

- 将价格、促销、排名、评论、内容、广告可见性、库存和渠道信号分别建模。
- 每个变化先检查采集口径、页面实验、地区、登录态和对象版本污染。
- 归因建立原因假设图，区分先行、同步和滞后信号，不以时间相邻证明因果。
- 阈值按平台、品类、生命周期和季节校准，并报告基线中位数与离散度。

### 证据与计算

- 标记 O/M/U/B/E/I，代理信号不得升级为竞品后台销量、花费、利润或库存事实。
- 指标输出公式、分子分母、标准化基准、模型版本和输入/输出哈希。
- 保留反对证据、未变化的关键面、替代解释和可能的采集失败。
- 跨模型版本必须同口径重算，禁止直接比较历史分数。

### 机会与反事实

- 优势判断同时检查来源、稀缺性、可复制性、持续性和我方准备度。
- 机会窗口必须有打开信号、关闭信号、最晚验证日和不行动代价。
- 比较跟随、差异化、观望和退出，不把竞品动作自动转成我方动作。
- 价格变化至少检验清仓、活动、变体切换、缺货和页面展示偏差。

### 失败与闭环

- 采集部分失败时保留已确认事实，范围和归因降级 inconclusive。
- 高置信归因需要至少三个一致且尽量独立的信号；否则给验证任务。
- 事件卡写明 owner、允许用途、禁止用途、责任域、成功、停止和复核日期。
- 下游动作由对应主权域决定；CIM只跟踪是否执行和结果是否支持原假设。
- 到期归因标记 validated/refuted/unresolved，并保留原版本与修订原因。

## product-identification-and-query-expansion 专属决策内核

| 维度 | 本模块要求 |
|---|---|
| 核心机制 | 从可见属性、用途、场景和替代方案形成候选对象与查询簇 |
| 计算或判定 | 置信度由独立属性匹配与冲突项决定 |
