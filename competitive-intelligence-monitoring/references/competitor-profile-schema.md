# 竞品画像与快照模式

## 核心对象

| 对象 | 必要字段 |
|---|---|
| 自有商品 | `own_product_id`、输入类型/来源、平台、市场、URL/商品 ID、品类、核心任务、客群、场景、价格带、关键属性、差异点、识别置信度、假设 |
| 竞品主体 | `competitor_id`、品牌/卖家、国家、渠道、经营背景 |
| 竞品商品 | `product_id`、`competitor_id`、平台、市场、URL、SKU/ASIN、品类、价格带、上架时间、发现查询词/渠道、纳入理由 |
| 监控快照 | `snapshot_at`、`product_id`、币种、价格、促销、排名、评分/评论、变体、库存信号、Listing/素材版本、来源 |
| 异动事件 | `event_id`、类型、前值/后值、幅度、持续期、证据、归因、置信度、影响 |
| 经营机会 | 对象、窗口类型/时长、建议动作、优先级、验证和停止条件 |

每个竞品商品只归属一个主体；每个快照只关联一个商品。新快照追加而不覆盖旧值。自有商品识别发生修订时保留原值、修订值、证据和时间，不静默覆盖。

## 画像维度

- 经营背景：主体、市场、渠道、经营时长和规模等级。
- 商品矩阵：SKU 宽度、价格带、主力/引流/利润款、上新节奏。
- 内容策略：主图、视频、A+/详情页、核心卖点与承诺。
- 评价结构：数量、增量、星级、差评主题和 QA。
- 运营节奏：促销、广告位、库存、社媒和上下架。

T1 画像完整度目标 ≥80%，T2 ≥60%。缺失字段必须保留为 `null` 并记录原因，不用推测值填补。

## 模块执行协议

### 对象与基线

- 锁定我方对象、竞品实体、商品/变体、店铺、平台、国家、展示条件和采集时间。
- 记录检索词、发现渠道、纳入/排除理由、覆盖停止条件和未覆盖面。
- 单次快照只能证明当时可见状态；趋势至少需要三个可比快照。

### 信号与机制

- 将价格、促销、排名、评论、内容、广告可见性、库存和渠道信号分别建模。
- 每个变化先检查采集口径、页面实验、地区、登录态和对象版本污染。
- 归因建立原因假设图，区分先行、同步和滞后信号，不以时间相邻证明因果。
- 阈值按平台、品类、生命周期和季节校准，并报告基线中位数与离散度。

### 证据与计算

- 标记 O/M/U/B/E/I，代理信号不得升级为竞品后台销量、花费、利润或库存事实。
- 指标输出公式、分子分母、标准化基准、模型版本和输入/输出哈希。
- 保留反对证据、未变化的关键面、替代解释和可能的采集失败。
- 跨模型版本必须同口径重算，禁止直接比较历史分数。

### 机会与反事实

- 优势判断同时检查来源、稀缺性、可复制性、持续性和我方准备度。
- 机会窗口必须有打开信号、关闭信号、最晚验证日和不行动代价。
- 比较跟随、差异化、观望和退出，不把竞品动作自动转成我方动作。
- 价格变化至少检验清仓、活动、变体切换、缺货和页面展示偏差。

### 失败与闭环

- 采集部分失败时保留已确认事实，范围和归因降级 inconclusive。
- 高置信归因需要至少三个一致且尽量独立的信号；否则给验证任务。
- 事件卡写明 owner、允许用途、禁止用途、责任域、成功、停止和复核日期。
- 下游动作由对应主权域决定；CIM只跟踪是否执行和结果是否支持原假设。
- 到期归因标记 validated/refuted/unresolved，并保留原版本与修订原因。

## competitor-profile-schema 专属决策内核

| 维度 | 本模块要求 |
|---|---|
| 核心机制 | 主体、品牌、店铺、商品、变体、快照、事件分实体建模 |
| 计算或判定 | 稳定ID与平台ID映射，属性带observed_at和来源 |
| 主要失效 | 店铺变化写商品、变体合并、同名品牌误合并 |
| 决策动作 | 身份置信不足不合并，保留候选边和拆分回滚 |

本节必须由该模块自己的证据和失败测试验证，不得用通用附录或其他reference代替。
