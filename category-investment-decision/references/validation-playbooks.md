# 测款与复盘剧本

## 原则

验证计划必须对应最弱假设，而不是机械跑完整流程。阈值按平台、国家、类目、账号历史和单位经济模型设置；没有历史基准时使用区间假设，并在首轮数据后校准。

只在平台允许的预售、众筹、落地页或广告形式中预验证需求。禁止虚假库存、伪造订单、无法履约的商品链接或违规收集用户数据。

Painted Door 或未上市产品必须在用户表达意向后明确告知尚未可买，不收取无法履约的款项。真实预售需要披露交付时间、退款条件和主要风险。

## 假设树与证据阶梯

测款前至少检查七类假设：需求、解决方案、信息/信任、价格、获客、产品/履约和单位经济。用“冲击×不确定性÷验证成本”排序，先测最可能推翻投资结论的假设。

| 证据层级 | 常见方法 | 主要能证明 | 不能单独证明 |
|---|---|---|---|
| 问题证据 | VOC、访谈、差评、退货原因 | 任务与痛点存在 | 愿意购买你的方案 |
| 概念证据 | 素材、概念页、假门 | 承诺能吸引注意 | 产品体验和可持续 CAC |
| 意向证据 | 留资、加购、开始结账 | 信息与价格产生较强意向 | 真实付款与使用价值 |
| 交易证据 | 合规预售、小批量销售 | 当前条件下的付款意愿 | 规模化获客和长期满意 |
| 使用证据 | 样品日记、留存、退货、复购 | 产品与履约价值 | 更大市场中必然成立 |

证据强度通常从表达、点击、留资、加购/结账、付款、使用到复购逐步提升。不把点击当成购买，不把小样本短期 ROAS 当成 PMF。

## 实验卡

每个实验事先记录：

1. 将被改变的商业决定；
2. 一个主假设和一个主指标；
3. 人群、渠道、排除条件和本轮唯一主要变量；
4. 基线、目标值、最小样本/时间窗口和数据来源；
5. `CONTINUE`、`ITERATE`、`STOP` 和 `INCONCLUSIVE` 条件；
6. 安全、合规、质量、现金流和误导红线；
7. 预算、周期、所需能力与最多迭代次数。

实验开始后不更换主指标或主门槛。同一轮尽量只改变一个核心变量。

## 四阶段验证

| 阶段 | 要验证什么 | 典型动作 | 通过后 |
|---|---|---|---|
| 1. 问题/概念 | 人群是否真有痛点并愿意付费 | 评论挖掘、访谈、问卷、落地页/素材概念测试 | 进入样品验证 |
| 2. 产品/供应 | 产品是否解决问题且能稳定交付 | 多供应商打样、盲测、质检、包装物流测试 | 小批量准备 |
| 3. 渠道/转化 | 平台上能否以可承受成本成交 | 自然流、多素材、达人微测、小预算广告 | 扩充样本与优化 |
| 4. 经济/复购 | 放量后是否仍盈利并产生复购 | 扩量、 cohort 复购、退货与现金周转跟踪 | 放量、迭代或清退 |

测试顺序可以因场景调整，但不得在产品、安全、合规或履约未验证时重仓放量。

## 指标与经济约束

| 环节 | 指标 | 判断方式 |
|---|---|---|
| 注意力 | 展示、3 秒留存、完播、CTR | 与同账号、同类目、同素材阶段比较 |
| 转化 | CVR、加购、CPA、ACOS、ROAS | 必须同时对照贡献利润和样本量 |
| 产品 | 盲测偏好、缺陷、评分、差评主题 | 区分产品问题、预期错配和履约问题 |
| 复购 | 30/60/90 天复购、订阅留存、配件续购 | 按产品合理使用周期选择窗口 |
| 经营 | 净贡献、退款、库存周转、现金回收期 | 使用实际结算和退货数据，不只看 GMV |

漏斗口径：`CTR = 点击/展示`，`到达率 = 落地页访问/点击`，`留资率 = 留资/落地页访问`，`加购率 = 加购/落地页访问`，`购买率 = 购买/落地页访问`，`退款率 = 退款/购买`，`CAC = 花费/购买`，`ROAS = 收入/花费`。分母为零时记为不可计算，不用 0 代替。

有结构化 JSON 输入时运行 `scripts/evaluate_experiment.py`，不手算多指标漏斗。工具只使用事先提供的最小样本和门槛，不自动发明行业健康线。

核心临界值从单位经济模型推导：

- Amazon：`实际 ACOS < 盈亏平衡 ACOS`，并为退货、促销和利润目标留安全垫。
- 内容/广告渠道：`实际 CPA < 扣除广告前单件贡献利润`；有可靠复购时才可使用 LTV 放宽。
- DTC：`盈亏平衡 ROAS = 1 ÷ 扣除广告前贡献利润率`；报告目标 ROAS 应高于盈亏平衡值。
- 所有平台：销量增长同时检查净贡献、退款和库存，避免“越卖越亏”。

## 进入、放量与停止

每个计划至少写出：

1. 待验证假设；
2. 样本或预算上限；
3. 观察周期；
4. 通过指标；
5. 停止条件；
6. 通过后的下一步。

结果判定顺序：

1. 最小样本/时间未达标 → `INCONCLUSIVE`；
2. 任一事先红线失败 → `STOP`；
3. 主门槛全部通过 → `CONTINUE`；
4. 最小证据已满足但主门槛未全通过 → `ITERATE`。

只在有明确可修正原因时迭代；达到事先限定次数仍不通过时停止。

出现以下任一情况，暂停放量并复核：

- 合规、IP、安全或平台禁限售红线；
- 压力场景亏损且没有已验证的修复路径；
- 供应商无法提供关键质量、追溯或交付证明；
- 转化只能依赖低于可盈利价格的持续折扣；
- 差评集中在产品本质缺陷，而非可修复的表达或履约问题；
- 补货周期长于剩余销售窗口，或现金占用超过卖家承受能力。

## 场景重点

- 趋势/季节：把到货时间、剩余销售周数、停止补货和清仓日期写入计划。
- Amazon：分开记录广告单与自然单，观察关键词排名和 Review 形成速度。
- TikTok：使用多钩子、多创作者的小样本矩阵，识别单条偶发爆量与可复制素材。
- DTC：先验证落地页和 CAC，再扩大库存；复购周期未到前不要假设 LTV。
- 店铺补品：验证新增 SKU 是否承接新流量或提升连带率，排除对现有 SKU 的纯蚕食。
- 老品延展：使用增量净贡献而非变体 GMV 判断成功。

## 复盘记录

每轮保留：场景、平台/国家、商品版本、素材版本、价格、流量来源、成本假设、实际指标、异常、结论和放弃原因。将结果标记为：`假设未验证 / 部分验证 / 已验证 / 被推翻`。

复盘后更新下一轮假设和阈值，但不要用单次成功案例改写全局评分标准。

复盘必须分开：事先假设、数据完整性、计算结果、实验能/不能证明的事、归因树、下一步和学习记录。失败时按需求、定向、素材、信息、价格、页面、付款、产品与履约分解，不直接归因为“没需求”。
